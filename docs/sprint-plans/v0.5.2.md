# v0.5.2 Sprint Plan

**Type**: Live-show operator polish release (post-v0.5.1)
**Timeline**: 1–2 weeks after v0.5.1 ships and stabilises in the field
**Branch strategy**: Each feature on its own `feature/*` branch off `master`

v0.5.0 shipped the major feature surface (MIDI Clock, Companion, Promo, Pre-Show, etc).
v0.5.1 shipped the build-system fix (Mac hardlink + Windows PowerShell). This v0.5.2 plan
focuses on operator pain points that don't need architecture changes.

## Premise

v0.5.2 addresses **live-show operator pain points** that can be fixed without architecture changes, plus collecting any regression bugs surfaced after v0.5.1 release in the field.

## In-scope features

### F1 — Pre-buffer next song (highest priority)

**Problem**: Switching songs today re-reads the file from disk + runs `decodeAudioData` synchronously during the Space/GO press. For large WAVs (>50 MB) this can freeze the UI for 1–2 seconds — unacceptable during a live show.

**Solution**: When `standbySetlistIndex` is set, pre-decode that file into a side buffer in the background. On GO, swap the decoded buffer instead of doing fresh I/O + decode.

**Scope**:
- `src/renderer/src/audio/AudioEngine.ts`
  - Add `prebufferFile(arrayBuffer: ArrayBuffer): Promise<AudioBuffer>` method that decodes without attaching to playback
  - Add `consumePrebuffered(): AudioBuffer | null` method that returns and clears the cached buffer
- `src/renderer/src/App.tsx`
  - Subscribe to `standbySetlistIndex` changes — kick off prebuffer fetch
  - When Space/GO fires, check for prebuffered buffer; if present, use it; otherwise fall back to existing `openFile` path
- `src/renderer/src/store.ts`
  - Add `prebufferingIndex: number | null` state (for UI feedback, optional)

**Non-goals**:
- Don't pre-decode the entire setlist (memory explosion on 50-song setlists)
- Don't prebuffer video files (out of scope)

**Acceptance criteria**:
- AC-1: Clicking a setlist song to put it in standby starts prebuffering in background within 100ms
- AC-2: Pressing Space to GO while prebuffer is complete transitions to playback in <100ms (visually instant)
- AC-3: Pressing Space before prebuffer completes falls back to existing behaviour (doesn't break)
- AC-4: Switching standby to a different song cancels the previous prebuffer
- AC-5: Prebuffered AudioBuffer is released when standby is cleared or file fails
- AC-6: No memory leak across 100 standby → GO cycles

**Estimated effort**: Small-medium (1 day)

### F2 — Offset scroll-wheel fine-tune

**Problem**: To nudge timecode offset by 1 frame, user must click the +/- buttons or drag the slider. On a pressured console, this is slow.

**Solution**: Mouse wheel over the `.tc-display` (or offset input) adjusts `offsetFrames` by ±1 per detent. Shift+wheel = ±10.

**Scope**:
- `src/renderer/src/components/Transport.tsx` — wheel handler on offset input
- `src/renderer/src/components/TimecodeDisplay.tsx` — wheel handler on TC digits

**Acceptance criteria**:
- AC-1: Scroll up over TC display increments offsetFrames by 1
- AC-2: Scroll down decrements by 1
- AC-3: Shift+scroll multiplies delta by 10
- AC-4: Respects existing clamp (-999..999)
- AC-5: Doesn't interfere with page/waveform scroll (preventDefault)

**Estimated effort**: Small (2 hours)

---

### F3 — OSC feedback / TC sync monitoring

**Problem**: In a complex show, the operator has no way to verify that MA3 / Resolume / Hog4 are actually receiving the timecode being sent. If a switch drops packets, the operator finds out when the lights freeze.

**Solution**: Accept incoming OSC "timecode ack" messages from downstream devices. Display received TC alongside sent TC in a small panel. Highlight drift >1 frame.

**Scope**:
- `src/main/index.ts` — OSC listen mode (new IPC `osc-feedback-start/stop`), UDP socket listening on configurable port
- Parse incoming OSC messages at `/ltcast/tc_ack` (address configurable)
- Forward received TC to renderer via IPC event
- `src/renderer/src/store.ts` — new `oscFeedbackDevices: Record<string, { lastTc, lastSeenAt }>`
- New small UI panel in Devices tab: "TC Feedback" showing each device + its last-reported TC + drift

**Acceptance criteria**:
- AC-1: Can enable OSC feedback in DevicePanel and set listen port
- AC-2: Multiple downstream devices can report in (identified by OSC source field or custom ID)
- AC-3: Panel shows sent vs received TC with colour-coded drift
- AC-4: Device disappears from panel if silent for >5 seconds
- AC-5: Works simultaneously with OSC output (different ports)

**Estimated effort**: Medium (1 day)

---

### F4 — Independent Show Timer

**Problem**: Current countdown is tied to the current audio file. There's no way to show "doors open in 15:00" or "intermission ends in 20:00" independent of playback.

**Solution**: Add a standalone show timer with multiple named presets. Displays prominently, optionally pipes to Art-Net/OSC for big-screen displays.

**Scope**:
- `src/renderer/src/components/ShowTimerPanel.tsx` — new component, new right-tab
- `src/renderer/src/store.ts` — timer state (target time, label, running)
- Persist across restart so "doors in 15 min" survives accidental app close
- Optional: broadcast timer value to OSC `/ltcast/show_timer/remaining` every second

**Acceptance criteria**:
- AC-1: Create named countdown: "Doors", "Intermission", "Lockout"
- AC-2: Timer runs independently of audio playback
- AC-3: Can be displayed fullscreen (for monitor audiences)
- AC-4: Optionally broadcasts to OSC

**Estimated effort**: Small-medium (half day)

---

### F5 — Stage Display mode upgrade

**Problem**: Current fullscreen TC shows only the timecode. An FOH engineer 10m from the operator needs to see song name, next song, remaining time — all at large scale.

**Solution**: Upgrade fullscreen TC to an info-rich stage display. Layout:
- Giant TC at top
- Current song name (medium)
- Remaining time + total duration (medium)
- "NEXT: [song name]" (small, only when autoAdvance on and last 60s)
- Optional: LTC signal dot + BPM indicator

**Scope**:
- `src/renderer/src/components/TimecodeDisplay.tsx` — extend `fullscreen` mode
- CSS for large-text layout
- All fields already exist in store; purely a layout change

**Acceptance criteria**:
- AC-1: Fullscreen shows TC + song + remaining + next
- AC-2: Text is legible from 10m (test with 1080p display)
- AC-3: No regression to non-fullscreen mode
- AC-4: Signal lost banner still reserves space (no layout shift)

**Estimated effort**: Small (half day)

---

### F6 — Setlist long-press drag

**Problem**: Drag-reorder on setlist activates on any mousedown. A nervous operator can accidentally re-order songs during a show.

**Solution**: Require 300ms hold before drag initiates. Visual cue (item lifts slightly) confirms drag mode entered.

**Scope**:
- `src/renderer/src/components/SetlistPanel.tsx` — mousedown timer, drag state

**Acceptance criteria**:
- AC-1: Quick click on setlist item does not initiate drag
- AC-2: Hold >300ms shows "drag armed" visual (e.g., subtle shadow/lift)
- AC-3: Drag then proceeds normally
- AC-4: UI Lock mode still blocks drag entirely

**Estimated effort**: Small (2 hours)

## Out of scope (deferred to v0.6.0)

- **Backup / Redundancy mode** — requires architecture for heartbeat protocol between two instances; own sprint
- **Audio Routing Matrix** — changes dual-AudioContext architecture; needs design spike
- **sACN (E1.31) support** — new protocol, needs packet-format module
- **SMPTE 2059 / PTP** — OS-level API (may need native binding); v0.7+
- **Marker-based GO list** — extends Standby/GO to be TC-triggered; design needed
- **Offline waveform cache (.ltcast-cache)** — performance, not blocking; v0.6

## Agent workflow

Per `CLAUDE.md` sprint rules:

1. **Planner** (read-only): expand this doc into per-feature `.claude/sprint-contract-F{N}.md` with concrete AC and file change lists
2. **Generator** (write): implements against the contract, produces `.claude/generator-report-F{N}.md`
3. **Evaluator** (read + run tests): independently verifies AC, produces `.claude/evaluator-report-F{N}.md`

Each feature runs its own A → B → C cycle. Features can be parallelised across branches as long as no file conflicts (F1/F5 touch TimecodeDisplay; don't run those concurrently).

Commits: no AI signatures (per project rule).
